Project #1: A data structure a day

Starting tommorow, I will implement a basic data-structure a day in C++ for the sake of learning the pointers and the language.

The first will be probably the stack, as it's the simplest. I will use OOP and test-based development extensively.Also, I will demonstrate the concepts of these here, in my blog.

I recommend this "let's do X a day"-method of learning, in the past, this made me learn new concepts and paradigms fast and relativily fun.If someone wants to follow me with this process in another language, say C# or Javasript, feel welcome, I would appreciate it.

A typical high-schoolers first question:
Why should I care?

You possibly have written programs which use arrays in the past. These give you the so called fast-random-access(you can say [10] and it gets the 10th element from it -many "programmers" take this for granted-), and have fast insertion at the end of the array. These properties are ideal for text-processing and many linear problems, where you need to append something to a list of values (most string types use array's as an internal container, so adding two strings together (concatenating) is as efficient as possible).
BUT:
What if I want to, say make a game, where I want to do the collision detection between the player and the objects around him/her?
Sure, I could just put every game object in an array and iterate through it, but this is a naive implementation and is O(n) complexity(For those who don't know: This is called Big-O-notation: where n is the game objects count, so n=10 takes x time and n=100 takes 10x time), and it can be enough to simple webdev, but not so for the programmer who gets hired at Google or Dropbox. He thinks "How could I make this more efficient?"; after thinking about the problem for a while he simplifies the problem to checking the small promixity of the player. This means we check the game objects place in the gamespace. This, again, is O(n), because we NEED to iterate over all of the elements, if we store them in an array.

Here comes our hero, The Almighty Quadtree! This in fact is a pretty complex data-structure, but you can see from the Wiki article, that it simplifies our problem to O(1) (constant time - in ideal circumstances), because we only need to check those elements whose are in the quad in which the player is located.

I got little bit carried away with this particular problem, but you can see, that learning data structures will pay off in the long run. See you tomorrow with the Stack!

Things needed for implementing data structures in C++:

Knowledge of OOP and pointers, and a good deal of tolerancy for SEGFAULT's :D
